SQL, Structured Query Language, is both a piece of server software which hosts and keeps organized a database system and a, well, structured language for interacting with it and querying the database.

That's a mouthful. An SQL database is essentially a collection of spreadsheets, which it calls tables. Tables have, just like spreadsheets, columns and rows. Tables are generally used to represent a list of objects.

Let's look at a table. This table's named Partiers.

FirstName LastName PhoneNumber Cool?
Tristán Peura 555-6477 True
Nathan Van Houten 555-8580 False
Vartolomej Dirkx 555-4881 True
Károly Janssens 555-5338 True
Kasper Kovačević 555-1720 False

As you can see, each row represents one of the objects in the list (in this case, a partier), and each column represents a property of those objects that, at least in theory, most objects will have a value for (otherwise, that row will have an empty, or "NULL" value, in that column). Each "cell" in spreadsheet terms, or individual piece of data (like Kasper's first name, or phone number) is called a field.

Now, SQL, like I said earlier, partly defines a language to interact with these tables. Let's check out a few commands in this language, with the caveat that there are multiple major flavors of SQL out there and that these commands might require slight tweaks for some flavors. This is just to give yall a taste of what interacting with SQL is like, moreso than an actual instructional primer on how to SQL. BEHOLD

Need to get some information from the table? Let's use a SELECT statement. I want to invite some people to a party. Let's get a list.

SELECT FirstName, PhoneNumber FROM Partiers WHERE Cool? = TRUE
FirstName PhoneNumber
Tristán 555-6477
Vartolomej 555-4881
Károly 555-5338

You might ask: "Why SQL?" Why didn't we settle on keeping all data that programs store in normal spreadsheets, or text files, or what have you? Why do we need all of this convoluted standard query syntax?

SQL is designed to be as efficient as is computationally possible.
Interacting through queries with a well-designed database - one with few, descriptive tables only occasionally needing to perform joins - is somewhere in the ballpark of an order of magnitude (ten times) faster performing than its non-SQL database serving cousins. So yeah. Use SQL.
