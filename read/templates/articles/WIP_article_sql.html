SQL, Structured Query Language, is both a piece of server software which hosts and keeps organized a database system and a, well, structured language for interacting with it and querying the database.

That's a mouthful. An SQL database is essentially a collection of spreadsheets, which it calls tables. Tables have, just like spreadsheets, columns and rows. Tables are generally used to represent a list of objects.

Let's look at a table. This table's named Partiers.

FirstName LastName PhoneNumber Cool?
Tristán Peura 555-6477 TRUE
Nathan Van Houten 555-8580 FALSE
Vartolomej Dirkx 555-4881 TRUE
Károly Janssens 555-5338 TRUE
Kasper Kovačević 555-1720 FALSE

As you can see, each row represents one of the objects in the list (in this case, a partier), and each column represents a property of those objects that, at least in theory, most objects will have a value for (otherwise, that row will have an empty, or "NULL" value, in that column). Each "cell" in spreadsheet terms, or individual piece of data (like Kasper's first name, or phone number) is called a field.

Now, SQL, like I said earlier, partly defines a language to interact with these tables. Let's check out a few commands in this language, with the caveat that there are multiple major flavors of SQL out there and that these commands might require slight tweaks for some flavors. This is just to give yall a taste of what interacting with SQL is like, moreso than an actual instructional primer on how to SQL. BEHOLD

Need to get some information from the table? Let's use a SELECT statement. I want to invite some people to a party. Let's get a guest list going, here.

SELECT FirstName, PhoneNumber
FROM Partiers
WHERE Cool? = TRUE

FirstName PhoneNumber
Tristán 555-6477
Vartolomej 555-4881
Károly 555-5338

(Don't worry about the spacing and line breaks in these queries. SQL, like most programming languages, doesn't really care what you do with either, so just try to make it look organized in a way that makes sense to you.) Anyway, here's what we get:

Let's break that down. SELECT is the command to retrieve data from a table, followed by the names of the columns you actually want the data from. FROM is necessary; you've got to specify the name of which table in the database you want the data from. The final bit we added there, the WHERE condition, isn't necessary for SELECT queries, but without it SELECT will just grab every row, so we use this to specify only to grab partiers whose "Cool?" value is TRUE. I don't need those uncool kids making messes all over my parents' house while they're out of town. I ain't cleanin' up after that.

Alright! Let's do something else. One of our closest homies is vouching for another partier. Let's throw em in the table. This partier's name is Pris Taylor, her number is 555-8905, and from what we hear she's cool. Let's put her in with another statement, an INSERT:

INSERT INTO Partiers
(FirstName, LastName, PhoneNumber, Cool?)
VALUES
( 'Pris', 'Taylor', '555-8905', TRUE)

Pretty straightforward, right? INSERT INTO >tablename<, the names of the columns, then specify the VALUES in the same order that you specified the columns. Right, well, theoretically she's in our growing rolodex now. Let's check:

SELECT * FROM Partiers

FirstName LastName PhoneNumber Cool?
Tristán Peura 555-6477 TRUE
Nathan Van Houten 555-8580 FALSE
Vartolomej Dirkx 555-4881 TRUE
Károly Janssens 555-5338 TRUE
Kasper Kovačević 555-1720 FALSE
Pris Taylor 555-8905 TRUE

Yep.

There are naturally more ways to interact with tables in SQL, but beyond adding (INSERT) and retrieving (SELECT) data, I'm only going to show you one more - the all-important JOIN. Excel nerds, these are your vlookups (basically). Non-Excel nerds, this is what you use to cross-reference tables. You'll see what I mean in a second.

Let's say we have another table, "PartyHouses". This is what that table looks like:

FirstName LastName Address
Tristán Peura 15 Rich Kid Ln
Nathan Van Houten 73 Irradiated Dr
Vartolomej Dirkx 2365 Backwoods St
Károly Janssens 800 Mugging Blvd
Kasper Kovačević 92 Gilded Ln
Pris Taylor 1 Bourgeois Ct

We're trying to find a spot for our next rager, but we want to be sure that it's at a cool person's house. We don't want them making a big deal out of the mess we totally leave there. Buckle up, this is about to get crazy:

SELECT Partiers.FirstName, Partiers.LastName,
Partiers.PhoneNumber, Partiers.Cool?,
PartyHouses.Address
FROM Partiers
JOIN PartyHouses ON Partiers.FirstName=PartyHouses.FirstName

Whoa! Let's break that down real quicklike. Firstly, you'll notice that since we're joining multiple tables, when specifying the columns we want to SELECT we need to specify which table each column is in using the format <tablename>.<columnname>. Not a huge deal. Which table you use as your FROM only matters when you're using specific types of JOINs I'll talk about in a second. Finally, we do the JOIN itself, which says that FirstName in the Partiers table is equal to FirstName in the PartyHouses table. In other words, we want to line up the data fields of the rows where the first name matches between the two tables. Phew. Here's what comes out:

FirstName LastName PhoneNumber Cool? Address
Tristán Peura 555-6477 TRUE 15 Rich Kid Ln
Nathan Van Houten 555-8580 FALSE 73 Irradiated Dr
Vartolomej Dirkx 555-4881 TRUE 2365 Backwoods St
Károly Janssens 555-5338 TRUE 800 Mugging Blvd
Kasper Kovačević 555-1720 FALSE 92 Gilded Ln
Pris Taylor 555-8905 TRUE 1 Bourgeois Ct

Boom. Now we have everything we need to figure out who to ring up.

You might ask: "Why SQL?" Why didn't we settle on keeping all data that programs store in normal spreadsheets, or text files, or what have you? Why do we need all of this convoluted standard query syntax?

SQL is designed to be as efficient as is computationally possible.
Interacting through queries with a well-designed database - one with few, descriptive tables only occasionally needing to perform joins - is somewhere in the ballpark of an order of magnitude (ten times) faster performing than its non-SQL database serving cousins. So yeah. Use SQL.
